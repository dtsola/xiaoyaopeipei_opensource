# 架构决策 - 对话记忆混合方案

**决策日期**：2026年02月25日

## 背景

小遥配配的C端AI对话功能需要实现多轮对话记忆，同时满足两个需求：

1. **AI上下文管理**：需要在对话过程中维护上下文，帮助AI理解用户意图
2. **商户端线索展示**：需要完整持久化所有对话消息，供商家查看客户咨询记录

## 解决方案

采用**混合记忆方案**：LangChain 记忆（内存缓存） + 数据库持久化

### 架构设计

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                       混合记忆架构                                            │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  ┌─────────────────────────────────────────────────────────────────────────┐ │
│  │  AI 层（LangChain 记忆）                                                 │ │
│  │  ├─ ConversationMemory: 内存缓存最近消息                                 │ │
│  │  ├─ max_messages=10: 最多保留 10 条消息                                  │ │
│  │  ├─ max_tokens=3000: 最多 3000 tokens                                   │ │
│  │  └─ 自动裁剪旧消息                                                       │ │
│  └─────────────────────────────────────────────────────────────────────────┘ │
│                                    ↕                                         │
│  ┌─────────────────────────────────────────────────────────────────────────┐ │
│  │  持久化层（数据库）                                                      │ │
│  │  ├─ messages 表: 保存所有消息（用于商户端线索展示）                       │ │
│  │  └─ load_from_db(): 从数据库加载历史到内存                              │ │
│  └─────────────────────────────────────────────────────────────────────────┘ │
│                                    ↕                                         │
│  ┌─────────────────────────────────────────────────────────────────────────┐ │
│  │  商户端线索展示                                                           │ │
│  │  └─ 从数据库读取完整对话历史                                              │ │
│  └─────────────────────────────────────────────────────────────────────────┘ │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 核心组件

**1. ConversationMemory（对话记忆）**
- 单个对话的记忆管理
- 支持懒加载（首次访问时从数据库加载）
- 自动裁剪（按消息数量和 token 数量）

**2. MemoryManager（记忆管理器）**
- 全局单例，管理多个对话的记忆
- 提供缓存和清理功能

### 实现细节

**文件位置**：[app/services/ai/memory.py](../xiaoyaopeipei-user-merchant-backend/app/services/ai/memory.py)

**使用示例**：
```python
# 获取对话记忆
memory = get_conversation_memory(
    conversation_id=conversation.id,
    db=db,
    max_messages=10,
    max_tokens=3000,
)

# 添加消息
memory.add_user_message("我需要一台游戏电脑")
memory.add_assistant_message("请问您的预算是多少？")

# 获取历史消息（用于 AI）
history = memory.get_dict_messages()
```

### 关键参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `max_messages` | 10 | 内存中最多保留的消息数量 |
| `max_tokens` | 3000 | 内存中最多保留的 token 数量 |

## 影响

### 优势

1. **性能提升**：内存缓存避免每次查询数据库
2. **Token 优化**：自动裁剪减少传入 LLM 的 token 消耗
3. **完整持久化**：所有消息保存到数据库，商户端可查看完整线索
4. **LangChain 兼容**：支持 LangChain 消息格式，便于后续扩展

### 依赖关系

- **新增依赖**：无（使用已有的 LangChain 组件）
- **修改文件**：
  - [chat.py](../xiaoyaopeipei-user-merchant-backend/app/api/user/chat.py) - 使用记忆功能

### 后续优化方向

1. 添加 Redis 缓存层（支持分布式部署）
2. 实现对话总结功能（当对话过长时）
3. 添加跨会话的长期记忆（用户画像）

## 决策理由

**为什么不纯用数据库？**
- 每次查询数据库影响性能
- 历史消息全部传给 LLM 消耗大量 token

**为什么不纯用内存？**
- 服务重启后丢失对话历史
- 商户端无法查看完整线索

**为什么选择混合方案？**
- 兼顾性能和持久化需求
- AI 上下文和线索展示分离，各取所需
- 使用 LangChain 标准接口，便于后续扩展
